# JeitinhoAéreo

> Um aplicativo de busca e reserva de voos com alma brasileira, construído como demonstração de design enraizado culturalmente e engenharia enxuta.

## Sobre o projeto

JeitinhoAéreo nasceu da ideia de que tecnologia boa não precisa ser fria e que produtos digitais feitos no Brasil podem carregar o Brasil no próprio nome sem pedir licença. A palavra *jeitinho* é praticamente intraduzível: descreve aquela forma criativa, amigável e descomplicada de resolver problemas que se tornou parte do vocabulário cotidiano brasileiro. Quando emparelhada com *aéreo*, o resultado é uma identidade que anuncia de imediato o que o aplicativo pretende ser — uma maneira prática e humana de comprar passagens, sem a frieza corporativa das grandes plataformas internacionais. O projeto foi construído em React com Vite, priorizando português como idioma padrão com suporte completo a inglês e espanhol, e estruturado para crescer ao longo do tempo sem virar caos.

## Objetivo

O objetivo deste repositório é duplo. Em primeiro lugar, serve como referência pedagógica para padrões modernos de desenvolvimento React: máquina de estados para fluxo de telas, Context API para internacionalização, componentes controlados com commit explícito para campos de autocomplete, SVG inline para bandeiras e ilustrações, separação clara entre dados, utilitários puros, primitivas de interface e componentes de tela, e uma disciplina cromática que reserva a cor de destaque para significado em vez de decoração. Em segundo lugar, o projeto serve como prova de conceito de que é possível entregar um produto visualmente polido e tecnicamente sólido sem recorrer a dependências pesadas ou frameworks externos para problemas que podem ser resolvidos em vinte ou trinta linhas de código bem escritas. A ambição é que cada arquivo deste repositório ensine algo concreto a quem o ler.

## Funcionalidades principais

O aplicativo oferece um fluxo completo de quatro telas que guiam o usuário desde a busca inicial até a confirmação da reserva. Na tela de busca, o usuário informa origem e destino através de campos com autocomplete em tempo real, sugerindo aeroportos à medida que digita e aceitando buscas por nome de cidade, código IATA ou país, com normalização de acentos para acomodar digitação apressada em dispositivos móveis. A lista contém mais de trinta aeroportos globais com prioridade para os principais aeroportos brasileiros. Após a busca, a tela de resultados apresenta uma lista de voos ordenados por preço, com a opção mais barata destacada automaticamente como *melhor oferta*, e simulação de atualização de preços a cada quatro segundos para reproduzir o comportamento de mercados reais de tarifas aéreas. A tela de reserva coleta dados de passageiro e permite escolha entre cinco formas de pagamento — cartão de crédito ou débito, Pix, Apple Pay, Google Pay e criptomoedas — com a ressalva honesta de que trata-se de um checkout de demonstração. A tela de confirmação encerra o fluxo com um código único prefixado por `JA-` que o usuário pode guardar. Em todas as telas, um seletor de idiomas no canto superior direito permite trocar entre português, inglês e espanhol instantaneamente, representado apenas por bandeiras SVG sem nomes de países, mantendo a interface limpa.

## Estrutura do projeto

O código-fonte vive em `src/`, organizado por tipo de responsabilidade. Em `src/data/` ficam os dados estáticos como a base de aeroportos, o catálogo de companhias aéreas e o dicionário de traduções para os três idiomas suportados. Em `src/lib/` ficam utilitários puros sem JSX, como a função de busca de aeroportos com normalização de acentos e o gerador determinístico de voos simulados. Em `src/context/` vive o provedor de idioma implementado com React Context, e em `src/hooks/` o hook customizado que consome esse contexto de forma segura em qualquer componente da árvore. Os componentes são subdivididos em três camadas dentro de `src/components/`: a pasta `ui/` contém primitivas reutilizáveis como campos de formulário e ícones de bandeira, a pasta `layout/` contém elementos que aparecem em múltiplas telas como o rodapé e o seletor de idiomas, e a pasta `screens/` contém os quatro componentes de tela que orquestram o fluxo do aplicativo. O arquivo `src/App.jsx` é o orquestrador que implementa a máquina de estados e amarra tudo.

A documentação arquitetural detalhada com justificativas de cada decisão vive em [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md), e o guia do sistema visual com regras de uso de cores, tipografia e espaçamento vive em [`docs/DESIGN_SYSTEM.md`](docs/DESIGN_SYSTEM.md). Essas são leituras recomendadas antes de contribuir com mudanças estruturais.

## Arquitetura e decisões técnicas

O gerenciamento de fluxo entre telas é feito por uma máquina de estados com quatro valores possíveis — `search`, `results`, `booking` e `done` — em vez de uma biblioteca de roteamento. Esta escolha é importante porque para um fluxo linear sem necessidade de deep linking, um roteador introduziria cerca de quarenta kilobytes de JavaScript e complicaria o modelo mental sem oferecer benefício real. A máquina de estados torna as transições imediatamente legíveis no código e evita estados inválidos por construção. Esta é uma aplicação prática de um princípio mais amplo que vale internalizar: prefira a ferramenta menos poderosa que resolve o problema real em vez da ferramenta mais poderosa que resolve problemas que você não tem.

A internacionalização é implementada com React Context e um dicionário de traduções estático, sem dependência de bibliotecas como i18next ou react-intl. O coração do sistema é uma função `t(chave)` que busca a tradução para o idioma atual, com fallback em cascata para inglês e depois para a própria chave caso a tradução esteja ausente — um padrão defensivo que garante que strings faltantes nunca causem crashes. Nomes de cidades e países vivem dentro dos próprios objetos de aeroporto em vez de serem chaves traduzíveis, refletindo uma distinção arquitetural importante entre dados com variantes linguísticas e textos de interface.

O autocomplete de aeroportos ilustra três padrões que aparecem em praticamente todo componente de dropdown maduro. O primeiro é o estado duplo: o input mantém seu próprio estado local enquanto o usuário digita, e só faz commit para o estado do pai quando uma sugestão é explicitamente selecionada, evitando que valores parciais ou inválidos vazem para a lógica da aplicação. O segundo é a detecção de clique externo através de um event listener no documento combinado com uma ref do componente, técnica que contorna o famoso conflito entre `onBlur` do input e `onClick` das sugestões. O terceiro é a normalização Unicode usando `NFD` seguida de remoção de marcas diacríticas, permitindo que o usuário encontre "Brasília" digitando "brasilia", detalhe especialmente importante para o público brasileiro.

As bandeiras do seletor de idiomas são todas SVGs inline de trinta por vinte e um pixels, proporção oficial 10:7 da maioria das bandeiras nacionais. Optamos por SVG em vez de emojis porque o suporte a emojis de bandeira é inconsistente entre sistemas operacionais, particularmente no Windows, e em vez de arquivos PNG porque SVGs carregam em bytes ao invés de kilobytes e escalam perfeitamente em qualquer resolução. As bandeiras são simplificadas intencionalmente — a bandeira americana não tem as cinquenta estrelas e a bandeira brasileira não tem a esfera celeste — porque no tamanho final em que são renderizadas, esses detalhes seriam ilegíveis e sugeririam precisão inexistente.

## Sistema de design

O sistema visual é construído sobre cores significativas definidas como variáveis CSS em `src/styles/globals.css`: navy `#2A1A4A` para superfícies escuras e chrome, laranja `#F58220` para valor e ação, ink `#1E1B3A` para texto principal, muted `#9A96B3` para texto secundário, e line `#EAE6F2` para divisores e bordas sutis. Esta disciplina cromática é o que separa um design que parece intencional de um que parece aleatório. O laranja aparece exclusivamente em preços, no botão principal de cada tela, no código do aeroporto destacado no autocomplete e no selo de melhor oferta — sempre onde existe significado real. Quando uma cor significa algo específico, o cérebro do usuário aprende rapidamente a escaneá-la; quando ela é decoração, vira ruído visual. A tipografia usa Plus Jakarta Sans em pesos de 400 a 800, carregada via Google Fonts, escolhida por sua personalidade mais distintiva que as opções padrão da web como Inter ou Roboto.

O layout em desktop envolve o aplicativo em uma moldura de telefone com cantos arredondados e sombra dupla simulando bisel físico. Esta escolha preserva a intenção mobile-first do design original em telas grandes, onde esticar uma interface projetada para 420 pixels até 1440 quebra proporções e destrói o ritmo espacial estabelecido pelo designer. Em dispositivos móveis reais, uma media query faz a moldura desaparecer e o aplicativo preenche a tela naturalmente.

## Como executar

O projeto requer Node.js versão 18 ou superior, conforme declarado em `.nvmrc`. Se você usa `nvm`, rode `nvm use` na raiz do projeto para garantir a versão correta. Após clonar o repositório, instale as dependências com `npm install`, e inicie o servidor de desenvolvimento com `npm run dev`. O Vite abrirá o aplicativo em `http://localhost:5173` com hot-reload ativo. Para gerar a build de produção, rode `npm run build`, que produzirá a pasta `dist/` pronta para ser servida por qualquer servidor de arquivos estáticos. Para verificar o código contra as regras de lint e formatação, rode `npm run lint` e `npm run format:check`. Nenhuma chave de API ou variável de ambiente é necessária para executar o protótipo, já que os dados de voos e a simulação de preços em tempo real são gerados localmente de forma determinística a partir dos parâmetros de busca.

Para publicar em produção, o guia [`DEPLOY.md`](DEPLOY.md) cobre as principais plataformas incluindo Vercel, Netlify, GitHub Pages, servidores próprios com Nginx, e deploy containerizado com Docker.

## Limitações conhecidas

Honestidade é parte do design deste projeto, então vale deixar claro o que o aplicativo não faz. A busca não consulta sistemas reais de distribuição de voos — as tarifas, companhias aéreas e horários são gerados localmente em `src/lib/flights.js` a partir de uma semente determinística. Para conectar a uma fonte real, seria necessário integrar com agregadores como Kiwi, Duffel ou Amadeus, transformando a função `generateFlights` em uma chamada assíncrona à API apropriada. O checkout de pagamento também é simulado; uma implementação real exigiria integração com um processador de pagamentos com conformidade PCI, tokenização de cartões no cliente e um backend para processar as transações com segurança. A persistência de estado é exclusivamente em memória, significando que recarregar a página apaga a reserva em andamento e a escolha de idioma volta ao padrão — adicionar `localStorage` ou um backend resolveria isso. Por fim, a lista de aeroportos em `src/data/airports.js` contém cerca de trinta e cinco entradas e teria que crescer consideravelmente para uso em produção, idealmente carregada de um banco de dados em vez de um array constante.

Atualmente, o projeto não possui suíte de testes automatizados — decisão aceitável em estágio de protótipo, mas que deve mudar conforme o escopo crescer. A recomendação documentada em `docs/ARCHITECTURE.md` é adotar Vitest para testes unitários de funções puras em `lib/`, React Testing Library para componentes críticos como o autocomplete, e eventualmente Playwright para testes end-to-end dos fluxos principais.
